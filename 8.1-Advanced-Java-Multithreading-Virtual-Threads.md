## Virtual threads

Original Java thread: `1 Java thread = 1 OS thread` (A simple wrapper)

Virtual threads are NOT OS threads
- Without them, a thread doing some I/O would always wait whenever it is assigned the CPU slots
- With them, a single OS thread (i.e carrier thread) can have multiple virtual threads scheduled (mounted) and unscheduled (unmounted)
  - Big benefit since now if a virtual thread is waiting on I/O on an OS thread occupying some time on a CPU, another virtual thread can be mounted on the OS thread in place of it i.e until it wakes up
- JVM manages the virtual threads, not the OS! That is a big plus! (Less context switching time)

**How to simulate slow I/O calls during learning?**
- Use **`Thread.sleep`**
- Later: We can use Spring Boot web to wait on an actual API request

## Alternate way to create an OS thread

Use `Thread thread = Thread.ofPlatform().unstarted(() -> Task.ioIntensive(finalI));` 

The `Thread.ofPlatform()` is a "builder"
- We can save the builder once and reuse it
- The builder can take a `name(<prefix>, <startcounter>)` for giving a "prefix" and a "count" for all our threads that we create

```java
private static void platformThreadDemo2() {
    // Only for demo purposes (Use thread pool & executor in production apps)
    Thread.Builder.OfPlatform builder = Thread.ofPlatform().name("push", 1);
    for (int i = 0; i < MAX_PLATFORM; i++) {
        int finalI = i; // Java needs it to be effectively final
        Thread thread = builder.unstarted(() -> Task.ioIntensive(finalI)); // This creates an OS thread
        // Same as: Thread thread = Thread.ofPlatform().name("push", 1).unstarted(() -> Task.ioIntensive(finalI)); 
        thread.start();
    }
}
```

Same result! CPU crashes if 1 million or so such OS threads are created

## Important consideration - Main thread waits

Even if the errors occur inside a thread or it takes too long, the main thread will not exit until the child thread finishes

Solution: Background thread! Use **`Thread.ofPlatform().daemon()`**

```java
private static void platformThreadDemo3() {
    // Only for demo purposes (Use thread pool & executor in production apps)
    Thread.Builder.OfPlatform builder = Thread.ofPlatform().daemon().name("daemon", 1);
    for (int i = 0; i < MAX_PLATFORM; i++) {
        int finalI = i; // Java needs it to be effectively final
        Thread thread = builder.unstarted(() -> Task.ioIntensive(finalI)); // This creates an OS thread
        thread.start();
    }
}
```

The code exits immediately (Unless we try to start a ton of threads like 1000s of them and OS stops providing us with memory and application crashes before it can start all the threads and exits)

## Making the main thread wait on daemon threads

Why? We will learn that later. For now, how can we make this happen?
1. Create a **"latch = countDownLatch(<number>)"** and pass in the number of threads you wish to wait on
2. Each child thread must indicate that it is done executing by calling the **`latch.countdown()`** method
2. Use **`latch.await()`** inside the main thread to wait for the child threads to finish

```java
package com.pushkar;

import java.util.concurrent.CountDownLatch;

public class IOTaskDemo {
    private static final int MAX_PLATFORM = 10;

    static void main() throws InterruptedException {
        platformThreadDemo4();
    }
    
    private static void platformThreadDemo4() throws InterruptedException {
        // Only for demo purposes (Use thread pool & executor in production apps)
        CountDownLatch latch = new CountDownLatch(MAX_PLATFORM);
        Thread.Builder.OfPlatform builder = Thread.ofPlatform().daemon().name("daemon", 1);
        for (int i = 0; i < MAX_PLATFORM; i++) {
            int finalI = i; // Java needs it to be effectively final
            Thread thread = builder.unstarted(() -> {
                Task.ioIntensive(finalI);
                latch.countDown();
            }); // This creates an OS thread
            thread.start();
        }
        latch.await();
    }
}
```

2. `Thread -> abstract BaseVirtualThread -> VirtualThread (private)`

Use **`Thread.ofVirtual()`** to instantiate virtual threads by using the builder!

## Scaling virtual threads

What is the benefit of virtual threads over platform threads?
- Since we have it managed by the JVM, it gets unmounted and mounted on a fixed number of platform threads
- **Benefit**: Hence, we can create 10, 100, 1000, 100000, or even millions of virtual threads and that creation will not crash the application!
- **Solution**: These virtual threads get interleaved by the JVM on existing platform threads (Not managed or created by OS)

```java
public class IOTaskDemo {
    private static final int MAX_VIRTUAL = 10_000_000;

    static void main() throws InterruptedException {
        platformThreadDemo5();
    }

    private static void platformThreadDemo5() throws InterruptedException {
        // Only for demo purposes (Use thread pool & executor in production apps)
        Thread.Builder.OfVirtual builder = Thread.ofVirtual(); // can use `var` too
        CountDownLatch latch = new CountDownLatch(MAX_VIRTUAL);
        for (int i = 0; i < MAX_VIRTUAL; i++) {
            int finalI = i; // Java needs it to be effectively final
            Thread thread = builder.unstarted(() -> {
                Task.ioIntensive(finalI);
                latch.countDown();
            }); // This creates an OS thread
            thread.start();
        }
        latch.await();
    }
}
```

## Virtual thread and stack

**"Stack chunk object"**: Virtual threads maintain a Task (Runnable) and a Stack within their object in the heap. This gets scheduled onto the carrier when mounted. The carrier stack manages the thread until it is unmounted, when it saves the stack back to the virtual thread and frees its own stack.

Mounting:
```
[ Heap ]               [ Carrier Thread ]
| VT 1 | --mount-->    | [VT 1 Stack]   |
|      |               | [VT 1 Task ]   |
```

Unmounting:
```
[ Heap ]               [ Carrier Thread ]
| VT 1 | <--unmount--  | (Empty / Idle) |
|      |               |      OR        |
|      |               | [VT 2 Stack]   |
```

Storage in the Heap
- Unlike platform threads, which have a fixed-size stack allocated by the OS, virtual threads are stored as objects in the Java heap.
- Task: This is the runnable code or lambda you want to execute.
- Stack: The execution state (frames, local variables) is stored as a "chunk" of data in the heap when not active.

<!-- TOC --><a name="what-is-a-forkjoinpool"></a>
## What is a ForkJoinPool

It is the pool of workers (OS thread pool) that are created and used to run virtual threads
- The pool tries to match the **number of CPU cores** when the number of virtual threads increases

<!-- TOC --><a name="cpu-vs-io-tasks-when-it-comes-to-virtual-vs-platform-threads"></a>
## CPU v/s IO tasks when it comes to Virtual v/s Platform threads

When choosing between Virtual Threads (VTs) and Platform Threads, the decision hinges on whether your application is waiting on the **CPU** or waiting on **External Resources**.

<!-- TOC --><a name="io-bound-tasks"></a>
### I/O-Bound Tasks

In I/O-bound tasks (network calls, database queries, disk access), the thread spends most of its time "waiting."

**Platform Thread (Heavy & Wasteful)**

The OS thread is tied to the task. While waiting for I/O, the thread is "blocked," but it still consumes a large memory footprint (e.g., 1MB stack) and keeps the OS scheduler busy.

```text
Thread: [---WORK---][.........WAITING FOR DB.........][---WORK---]
OS Res: [================== RESERVED ============================]
```

**Virtual Thread (Light & Efficient)**

When the VT hits a blocking I/O call, it **unmounts**. The carrier thread is released to process other VTs. The VT only consumes a small amount of space in the Heap while waiting.

```text
VT 1:   [--WORK--] (Unmounts)  ........WAIT........  (Remounts) [--WORK--]
Carrier:          [--VT 2--][--VT 3--][--VT 4--][--VT 2--] 
```

**Winner:** **Virtual Threads**. They allow "Thread-per-request" style code to scale to millions of concurrent I/O operations.

<!-- TOC --><a name="cpu-bound-tasks"></a>
### CPU-Bound Tasks

In CPU-bound tasks (video encoding, complex math, heavy data processing), the thread is constantly using the processor. There is no "waiting" period.

**Platform Thread (Direct)**

The task runs directly on the OS thread, utilizing the CPU core fully.

```text
CPU Core: [################### PROCESSING ####################]
Thread:   [################### PROCESSING ####################]
```

**Virtual Thread (Overhead)**

Because the CPU is always busy, the VT never "unmounts" to yield. You gain no concurrency benefits, and you actually lose a tiny bit of performance due to the JVM scheduling overhead.

```text
CPU Core: [################### PROCESSING ####################]
Carrier:  [################### PROCESSING ####################]
VT:       [################### PROCESSING ####################]
             ^ JVM Overhead: managing the VT adds no value here.
```

**Winner:** **Platform Threads**. For pure computation, you want as little abstraction as possible between your code and the hardware.

<!-- TOC --><a name="summary-which-one-is-better-when"></a>
### Summary: Which one is better when?

| Scenario | Recommended Thread Type | Reason |
| :--- | :--- | :--- |
| **Microservices / Web APIs** | **Virtual** | High I/O (waiting for DB/other services). |
| **Database Clients** | **Virtual** | Most time is spent waiting for the wire. |
| **Video/Image Processing** | **Platform** | Constant CPU usage; no blocking to benefit from. |
| **Cryptography/Hashing** | **Platform** | Intensive CPU work. |
| **High Concurrency (>10k)** | **Virtual** | OS threads would crash the system due to memory exhaustion. |


<!-- TOC --><a name="the-golden-rule"></a>
### The Golden Rule
* Use **Virtual Threads** for **Scaling**: When you have many tasks that spend time waiting.
* Use **Platform Threads** for **Throughput**: When you have a few tasks that need every ounce of CPU power.

<!-- TOC --><a name="what-if-we-really-wanted-to-use-virtual-threads-for-cpu-intensive-tasks"></a>
## What if we really wanted to use Virtual Threads for CPU intensive tasks?

Platform threads are better for CPU intensive tasks!
- Priority can be set in order to make sure OS can schedule accoringly (1 low, 10 high)
- However, virtual threads are all defaulted to 5 priority - cannot be changed!

If you still want to use Virtual Threads for CPU intensive tasks, you will face the following issue:
- One virtual thread does a CPU intensive task and OS cannot schedule it out
- Since all have the same priorities (5), there is no context switch - ON THE SAME CPU
- So, if you want to give another starving Virtual Thread a chance to execute on the same CPU core then the current one must "yield"
  - **`Thread.yield()`**: Unmounts the current task and mounts another. Re-mounts the original task when CPU becomes free
    - For platform threads, the OS scheduler might "ignore" the `yield()` (Quite common)
    - For virtual threads, the JVM is the scheduler so it never really ignores it (Works precisely for virtual threads)

<!-- TOC --><a name="how-virtual-threads-can-help-practical-use-cases"></a>
## How virtual threads can help (Practical use cases)

**Virtual threads are extremely useful in I/O -> Especially microservices calling other microservices and waiting on them!**

In a traditional microservice setup, your app acts like a middleman. It calls Service A, waits, then calls Service B. If you use standard **Platform Threads**, that thread is "stuck" doing nothing while waiting for the network.

<!-- TOC --><a name="the-old-way-platform-threads"></a>
### The "Old" Way (Platform Threads)
Each request takes up one expensive OS thread. If the Payment service is slow, that thread is held hostage.

```text
[ Request 1 ] --> [ Thread A (BLOCKED) ] --waiting for Payment--> [ Payment Svc ]
[ Request 2 ] --> [ Thread B (BLOCKED) ] --waiting for Billing--> [ Billing Svc ]
[ Request 3 ] --> [ Thread C (WAITING) ] --No threads left in pool!--
```

**Problem:** You run out of threads quickly, and your server crashes even though the CPU isn't actually doing any work.

<!-- TOC --><a name="the-new-way-virtual-threads"></a>
### The "New" Way (Virtual Threads)
When your code calls a microservice, the Virtual Thread **unmounts**. The physical worker (Carrier Thread) immediately moves to the next request.

```text
[ Heap Storage ]         [ Carrier Thread (The Worker) ]
| VT-Request 1 | (Idle)  | [ VT-Request 3 ] (Processing) |
| VT-Request 2 | (Idle)  |                               |
```

<!-- TOC --><a name="code-transformation"></a>
### Code Transformation
Here is a typical method calling three microservices. Wrapping this in a Virtual Thread is as simple as using a new executor.

**The Method:**
```java
public void processOrder(Order order) {
    // These are I/O calls (Blocking)
    paymentSvc.charge(order); // Blocked on this I/O first
    billingSvc.generateInvoice(order); // Next: Blocked on this I/O 
    shippingSvc.arrangeDelivery(order); // Last: Blocked on this I/O
}
```

**The Implementation:**
```java
// Create an executor that spawns a new Virtual Thread for every task
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> processOrder(myOrder)); // When virtual thread is blocked, it is unmounted 
    // ... and another virtual thread is mounted on the carrier OS thread, if available
}
```

<!-- TOC --><a name="why-is-this-better"></a>
### Why is this better?

1.  **Massive Concurrency:** You can now have 100,000 orders processing at the exact same time. On Platform Threads, you'd likely hit a limit at 200–500 before the memory runs out.
2.  **Code Simplicity:** You are writing **Sequential Code**. You don't need complex "Reactive" libraries (like WebFlux) or "Callbacks." It looks like old-fashioned Java but performs like high-end asynchronous code.
3.  **Resource Efficiency:** * **Memory:** Virtual threads are tiny (kilobytes) compared to Platform threads (megabytes).
    * **CPU:** The Carrier Thread stays busy switching between active tasks instead of sitting idle waiting for a network response from the Shipping service.

<!-- TOC --><a name="summary-io-vs-cpu-in-this-example"></a>
### Summary: I/O vs. CPU in this example
Since calling `paymentSvc`, `billingSvc`, and `shippingSvc` are all **I/O tasks** (sending data over a wire), the Virtual Thread will ***automatically yield*** (hand over) the CPU every time it waits for a response. This is the "Cooperative" behavior we discussed—no manual `yield()` needed!

<!-- TOC --><a name="synchronization-problems-with-virtual-threads"></a>
## Synchronization problems with Virtual Threads

We still have all the issues of race conditions, data race, deadlock, and so on even with Virtual Threads (Nothing changes!)

<!-- TOC --><a name="thread-pinning-problem-wrt-virtual-threads"></a>
### Thread pinning problem w.r.t Virtual Threads

When you use **Virtual Threads (VT)** in Java 21, 22, or 23, they generally play nice by "unmounting" from the Carrier Thread (the physical worker) whenever they hit a blocking I/O task. 

However, there is a "trap" called **Thread Pinning**.

**The Problem: Thread Pinning (J21, J22, J23)**

In these versions, if a Virtual Thread enters a `synchronized` block or method and then hits a blocking operation (like a database call), it becomes **pinned** to the Carrier Thread. 

Instead of unmounting to let other VTs run, it "gets stuck" to the Carrier. If all your Carrier Threads get pinned, your entire application stops—even if you have millions of Virtual Threads available.

```text
[ Carrier Thread (Physical) ]
      |
      |  <-- VT enters 'synchronized'
      |
[ VT-1 (PINNED) ] <--- Waiting for DB response...
      |
      |  <-- CANNOT UNMOUNT! 
      |      Carrier is now useless to VT-2, VT-3, etc.
```

**The Solution: ReentrantLock (The J21-23 Workaround)**

To prevent pinning in older versions, you replace `synchronized` with `ReentrantLock`. Unlike `synchronized`, `ReentrantLock` allows the Virtual Thread to unmount successfully even if it's holding the lock.

```java
import java.util.concurrent.locks.ReentrantLock;

public class PaymentService {
    // 1. Define a ReentrantLock instead of using 'synchronized'
    private final ReentrantLock lock = new ReentrantLock();

    public void processPayment() {
        lock.lock(); // 2. Acquire the lock
        try {
            // 3. Perform I/O (e.g., calling a microservice)
            // In J21-23, the VT will UNMOUNT here perfectly.
            callExternalAPI(); 
        } finally {
            lock.unlock(); // 4. Always release in finally block
        }
    }
}
```

**How Java 24 Fixed It**

**Java 24** (via JEP 491) completely changed the game. The JVM engineers rewrote how `synchronized` works internally.

Now, in Java 24, a Virtual Thread can unmount even while inside a `synchronized` block. You no longer have to worry about "pinning" or manually switching everything to `ReentrantLock`.

<!-- TOC --><a name="ascii-diagram-java-24-behavior"></a>
### ASCII Diagram: Java 24 Behavior
```text
[ Carrier Thread ]        [ Heap ]
       |                     |
       |  -- Unmount -->     | [ VT-1 (Holding Monitor Lock) ]
       |                     |   (Waiting for DB...)
       |                     |
[ VT-2 (Starts) ]            |
 (New worker)                |
```

| Java Version | `synchronized` + I/O | `ReentrantLock` + I/O |
| :--- | :--- | :--- |
| **J21, J22, J23** | **Pinned** (Bad for scaling) | **Unmounts** (Good) |
| **J24+** | **Unmounts** (Fixed!) | **Unmounts** (Good) |

**Key Takeaway**
* If you are on **Java 21-23**, pinning is a major risk in legacy code using `synchronized`. The fix is migrating to `ReentrantLock`.
* If you are on **Java 24**, the "Virtual Thread Pinning" issue is mostly a thing of the past for monitors, making it much easier to adopt VTs in older codebases.

<!-- TOC --><a name="thread-safety-with-thread-builder"></a>
## Thread safety with Thread Builder

- "Thread builder" is **NOT** thread-safe!
- It usually ONLY MATTERS when you create child threads and they in turn create their own child threads (Ex: An external lib creating threads you use inside your own thread). That is, a hierarchy of threads being created
- It is better to make the "Thread builder" thread-safe by using the **`.factory()`** & the **`factory.newThread(<runnable>)`** methods in such a case

To make virtual threads "thread-safe" in this context, we aren't talking about protecting a single variable. Instead, we are talking about **Thread Confinement**: ensuring that every task gets its own dedicated, fresh thread so they don't step on each other’s toes.

Using a **Thread Factory** is the professional way to "template" how these threads are created.

<!-- TOC --><a name="using-the-builder-to-create-a-factory"></a>
### Using the Builder to Create a Factory

Java provides a fluent **Builder API** to configure virtual threads. You can set a name prefix (great for debugging) and then store this configuration in a `ThreadFactory`.

```java
import java.lang.Thread.Builder;
import java.util.concurrent.ThreadFactory;

public class VirtualThreadSafeFactory {
    public static void main(String[] args) {
        
        // 1. Use the Builder to create a Virtual Thread Factory
        ThreadFactory factory = Thread.ofVirtual()
                .name("payment-worker-", 0) // Names: payment-worker-0, payment-worker-1...
                .factory();

        // 2. Use the factory to create a NEW thread for a task
        // This ensures the task has its own isolated stack (Thread Confinement)
        Thread t1 = factory.newThread(() -> {
            System.out.println("Processing in: " + Thread.currentThread().getName());
        });

        t1.start();
    }
}
```

<!-- TOC --><a name="why-a-factory-ensures-safety-thread-confinement"></a>
### Why a Factory ensures "Safety" (Thread Confinement)

In traditional thread pools, threads are **reused**. If you forget to clear a `ThreadLocal` variable, the next task might "see" data from the previous user. 

With a Virtual Thread Factory, the mental model shifts: **Threads are disposable.**

```text
[ Factory ] 
    |
    |--> [ VT-0 ] (Task A) --> [ Finish ] --> [ Destroyed ]
    |
    |--> [ VT-1 ] (Task B) --> [ Finish ] --> [ Destroyed ]
    |
    |--> [ VT-2 ] (Task C) --> [ Finish ] --> [ Destroyed ]
```
**Safety Benefit:** Because the thread is destroyed after one use, there is no "leftover" state. It provides a clean slate for every single request.

<!-- TOC --><a name="why-executors-are-still-the-better-method"></a>
### Why Executors are STILL the better method

While the `ThreadFactory` is cool, you should almost always use the **`ExecutorService`** (specifically the `newVirtualThreadPerTaskExecutor`) instead of manually calling `factory.newThread()`.

**Reason 1: The "Auto-Close" Pattern (Structured Concurrency)**

Executors support the `try-with-resources` block. This ensures that your main program waits for all virtual threads to finish before moving on.
```java
// BEST PRACTICE
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(task1);
    executor.submit(task2);
} // <--- All threads are guaranteed to be finished here!
```

**Reason 2: Future & Result Handling**

If you use `factory.newThread()`, getting a return value back is hard (you have to use shared variables). 
With an **Executor**, you get a `Future<T>`, which is the standard way to handle results and exceptions in Java.

**Reason 3: Abstraction**

Using an `ExecutorService` means your business logic doesn't care if the threads are Virtual or Platform. You can swap the underlying thread type in one line of code without touching your task logic.

<!-- TOC --><a name="comparison-summary"></a>
### Comparison Summary

| Feature | Manual Factory (`newThread`) | Executor Service |
| :--- | :--- | :--- |
| **Complexity** | High (you must manage `.start()`) | Low (just `submit()`) |
| **Clean up** | Manual | Automatic (with try-with-resources) |
| **Return Values** | Difficult | Easy (via `Future`) |
| **Standard** | No | **Yes (Industry Standard)** |

**Bottom Line:** Use the **Builder** to understand how threads are configured, but use the **Executor** to actually run your microservice tasks. It’s safer, cleaner, and handles the "waiting" for you.

<!-- TOC --><a name="executor-service-production-ready-way-to-create-threads"></a>
## Executor Service - Production ready way to create threads

**ExecutorService** - What is it?
- It is a high level *concurrency framework*
- It does the following:
  1. Thread managment
  2. Tasks handling

A virtual thread can be thought of as a "sub-task"
- Imagine an airline price check task
- If it needs to check the prices on 10 different airlines to get the best price, checking each would become a "sub-task"
- Each of these sub-tasks can be a virtual thread (Since VTs are very cheap)

An executor service will help with this!

<!-- TOC --><a name="terminology"></a>
### Terminology

Think of the **Executor** framework as a "Hiring Agency" for your code. Instead of you manually managing workers (threads), you just hand over the task to the agency and they handle the logistics.

**1. Executor (The Simple "Do It" Interface)**

This is the most basic level. It has only one job: **Execute a task.** It doesn't promise to tell you when it's done or give you a result; it just takes the task and runs it.

* **Layman Definition:** A mailbox where you drop a letter, and someone eventually picks it up to deliver it.
* **Functional Interface:** Since it has only one method (`execute`), you can use a Lambda.

```java
// Executor: "Just run this, I don't need an update."
Executor simpleRunner = (Runnable task) -> new Thread(task).start();
simpleRunner.execute(() -> System.out.println("Task is running!"));
```


**2. ExecutorService (The "Manager" Interface)**

This is a more advanced version of the Executor. It doesn't just run tasks; it **manages** them. It can track if a task is finished, return a result (via a `Future`), and shut down the whole operation safely.

* **Layman Definition:** A project manager. You give them a job, and they give you a "tracking number" so you can check the status later.

```java
// ExecutorService: "Run this and give me a receipt (Future)."
ExecutorService manager = ...; // Created using the Util class below
Future<String> receipt = manager.submit(() -> {
    Thread.sleep(1000);
    return "Task Finished!";
});

// You can check the receipt later
if (receipt.isDone()) {
    System.out.println(receipt.get());
}
```

**3. Executors (The "Utility" Factory)**

This is a helper class (a "Utility" class). You don't "run" anything with it; instead, you use it to **create** different types of managers (`ExecutorService` implementations).

* **Layman Definition:** The catalog you look through to choose which kind of hiring agency you want (e.g., a "Fixed Team of 5" or "As Many Workers as Needed").

```java
// Executors: "Give me a manager with 10 fixed threads."
ExecutorService fixedPool = Executors.newFixedThreadPool(10);

// Executors: "Give me a manager that uses Virtual Threads."
ExecutorService vtPool = Executors.newVirtualThreadPerTaskExecutor();
```

```text
   [ Executors (Util Class) ] 
             |
             |-- "Hey, give me an Implementation!"
             v
   [ ExecutorService (Interface) ]  <-- (The Manager: submit(), shutdown())
             |
             |-- "I extend..."
             v
   [ Executor (Interface) ]         <-- (The Worker: execute())
```

<!-- TOC --><a name="summary-table"></a>
#### Summary Table

| Name | Type | Purpose | Analogy |
| :--- | :--- | :--- | :--- |
| **Executor** | Interface | The bare minimum to run a task. | A "Push" button. |
| **ExecutorService** | Interface | Manages life cycle and results. | A Service Desk with tracking. |
| **Executors** | Class | Factory methods to build the above. | The Factory/Store. |

<!-- TOC --><a name="why-this-is-the-better-method"></a>
#### Why this is the "Better Method"
Instead of manually creating a `Thread factory` and calling `.newThread().start()` for every task, you just use **`Executors.newVirtualThreadPerTaskExecutor()`**. 

1.  **Managed Shutdown:** It cleans up all threads when done.
2.  **Standardized:** Every Java developer understands an `ExecutorService`.
3.  **Result Handling:** It’s the only clean way to get data back from a thread using `Future`.

<!-- TOC --><a name="different-executors-for-regular-threads"></a>
## Different executors for regular threads

Think of the **Executors** utility class as a "Hiring Agency" that provides different types of specialized teams. Depending on your workload (CPU vs. I/O) and how much memory you have, you pick the right "team" for the job.

<!-- TOC --><a name="1-fixed-thread-pool"></a>
### 1. Fixed Thread Pool
This is a team with a **fixed number of workers**. If all workers are busy, new tasks wait in a queue (like a waiting room) until someone finishes.

* **Best for:** Resource management. You know exactly how many threads your CPU can handle.
* **Analogy:** A coffee shop with exactly 3 baristas. If 10 people come in, the 4th person waits until a barista is free.

A Spring application uses this and has 200 threads by default!

```java
// Create a pool with exactly 3 threads
ExecutorService fixedPool = Executors.newFixedThreadPool(3);

fixedPool.submit(() -> System.out.println("Task 1"));
fixedPool.submit(() -> System.out.println("Task 2"));
// Task 4 would wait in the LinkedBlockingQueue
```

<!-- TOC --><a name="2-single-thread-executor"></a>
### 2. Single Thread Executor
This is a team of **exactly one worker**. Tasks are guaranteed to be executed one after another (sequentially).

* **Best for:** Tasks that must happen in order or shouldn't run at the same time (e.g., writing to a specific log file).
* **Analogy:** A single-lane drive-thru. No matter how many cars arrive, they are served one by one.

```java
// Only one thread exists
ExecutorService singlePool = Executors.newSingleThreadExecutor();

singlePool.submit(() -> System.out.println("First"));
singlePool.submit(() -> System.out.println("Second")); // Guaranteed to run after 'First'
```

<!-- TOC --><a name="3-cached-thread-pool"></a>
### 3. Cached Thread Pool
This is an **elastic team**. It creates new threads as needed but reuses old ones if they are free. If a thread sits idle for `60` seconds, it is terminated to save memory.

* **Best for:** Many short-lived asynchronous tasks. **Warning:** Can crash your system if you submit too many tasks at once because it keeps spawning threads!
* **Analogy:** A taxi company. They don't have a fixed number of cars; they just send more out as calls come in, and drivers go home when it's quiet.

Note: It starts with `0` threads and based on the number of task submissions, it will create a new thread for each & reuse the created ones whenever possible (i.e one thread finishes and makes a thread become available)

```java
// No fixed size; grows and shrinks dynamically
ExecutorService cachedPool = Executors.newCachedThreadPool();

cachedPool.submit(() -> System.out.println("Task A"));
```

<!-- TOC --><a name="4-scheduled-thread-pool"></a>
### 4. Scheduled Thread Pool
A team that can run tasks **after a delay** or **repeatedly** at fixed intervals.

* **Best for:** Heartbeats, clean-up tasks, or sending "delayed" notifications.
* **Analogy:** An alarm clock or a sprinkler system that turns on every day at 6:00 PM.

```java
// Create a pool that supports scheduling
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);

// Run after 5 seconds
scheduler.schedule(() -> System.out.println("Delayed Task"), 5, TimeUnit.SECONDS);

// Run every 10 minutes
scheduler.scheduleAtFixedRate(() -> System.out.println("Beep"), 0, 10, TimeUnit.MINUTES);
```


```text
TYPE             | THREADS | QUEUE TYPE      | BEST FOR
-----------------|---------|-----------------|-------------------------
Fixed            | N       | Linked (Bound)  | Steady, predictable load
Single           | 1       | Linked (Bound)  | Sequential ordering
Cached           | 0 to ∞  | Synchronous     | Short, bursty tasks
Scheduled        | N       | Delayed Queue   | Timers / Repeating tasks
Virtual (New!)   | ∞ (VTs) | N/A             | High I/O / Scaling
```

<!-- TOC --><a name="which-one-for-virtual-threads"></a>
### Which one for Virtual Threads?
Remember: **Virtual Threads don't need "Pools."** Because they are so cheap, we use `Executors.newVirtualThreadPerTaskExecutor()`. 
- We don't limit the number of virtual threads (pooling) because the whole point is to have as many as we need!

<!-- TOC --><a name="virtual-threads-are-not-supposed-to-be-pooled"></a>
## Virtual threads are NOT supposed to be pooled!

Reasons:
- Virtual thread is supposed to be for a "task" or "sub-task"
- Hence, they are intended to be **created on demand** and **discarded once the task is done**

Then, should we *really* use the ExecutorService with Virtual Threads?
- We can!: There is a model called **"Thread Per Task"** creation management and Virtual Threads are recommended to use this! (only)
- Available from **Java 21** only

This is the most important "mindset shift" for Java 21+. In the old world, threads were expensive **Resources** to be saved. In the new world, Virtual Threads are **Tasks** to be thrown away.

**Why Pooling is "Bad" for Virtual Threads**

A **Platform Thread** is like a **Luxury Car**: You only have a few, they are expensive to maintain, so you keep them in a "Garage" (Pool) and reuse them for different drivers.

A **Virtual Thread** is like a **Disposable Napkin**: You use it once for a specific mess (task) and throw it away. Attempting to "pool" them is like trying to wash and reuse a paper napkin—it’s more work than just getting a new one.

Pooling vs. Thread-Per-Task:
```text
[ OLD WAY: POOLING ]              [ NEW WAY: THREAD-PER-TASK ]
      (Reuse)                            (Create & Discard)
   Thread Pool [3]                    Task A --> [ New VT ] --> Done
    /     |     \                     Task B --> [ New VT ] --> Done
 TaskA  TaskB  TaskC                  Task C --> [ New VT ] --> Done
```

**The "Thread-Per-Task" Model**

Because Virtual Threads are so light (kilobytes of memory), Java 21 introduced a specific `ExecutorService` implementation designed **only** to create a fresh thread for every single task and destroy it immediately after.

Instead of picking a size (like 10 or 100), you just tell Java: "Give me a new Virtual Thread for every job I submit.":
```java
import java.util.concurrent.Executors;

public class TaskManager {
    public void handleRequests() {
        // 1. Use the specific "Thread-Per-Task" Executor
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            
            for (int i = 0; i < 1000; i++) {
                int taskId = i;
                // 2. This creates a BRAND NEW Virtual Thread
                executor.submit(() -> {
                    System.out.println("Processing task " + taskId + 
                                       " on " + Thread.currentThread());
                    // Task finishes, VT is discarded automatically!
                });
            }
            
        } // 3. Executor closes here, waiting for all 1000 VTs to finish
    }
}
```

**3. Why we *still* use ExecutorService**

If we aren't "pooling," why not just call `Thread.startVirtualThread()`?

1.  **Structured Concurrency:** As shown above, the `try-with-resources` block acts as a barrier. Your code won't move past the `}` until **all** submitted tasks are done.
2.  **Compatibility:** Most existing Java libraries (Spring, Hibernate, etc.) expect an `ExecutorService`. By using `newVirtualThreadPerTaskExecutor()`, you can upgrade your app to handle millions of requests just by changing one line of code.
3.  **Future-Proofing:** It gives you a handle on your tasks (`Future`) so you can cancel them or check if they failed, which is much harder to do with raw `Thread` objects.

Summary Checklist:
* **Do I need to limit the number of VTs?** No. Let the JVM handle the millions.
* **Do I reuse VTs?** No. Let them die when the task is done.
* **How do I run them?** Always prefer `Executors.newVirtualThreadPerTaskExecutor()`.

<!-- TOC --><a name="executor-service-methods-to-know"></a>
## Executor service methods to know

Think of the **ExecutorService (ES)** as a professional department in a company. Even if there is no work to do, the department stays "open" and the employees stay on the payroll until the CEO officially closes the office.

<!-- TOC --><a name="submit"></a>
### `submit()`
This is how you give a task to the department. When you call `submit()`, the ES takes the task and gives you a **Future** (a receipt).

**The "Keep-Alive" Behavior:**
Even after the task is finished, the application will **not** exit. This is because the threads inside the ES are "Non-Daemon" threads—they are told to stay alive and wait for more work.
* **Analogy:** A waiter at a restaurant doesn't go home just because your table finished eating; they stay until the restaurant closes.

```text
[ Task ] --submit()--> [ Executor Service ] --> [ Result ]
                           (Thread stays alive)
```

<!-- TOC --><a name="shutdown"></a>
### `shutdown()`
This is the "Polite Close." It tells the ES: "Finish what you are currently doing, but don't accept any new tasks."

* **Behavior:** It stops accepting new `submit()` calls (which will throw an exception). It waits for all currently running and queued tasks to finish.
* **Analogy:** Turning off the "Open" sign at a shop. People already inside can finish shopping, but no one else can enter.

```text
[ Task 1 (Running) ]  <-- Will finish
[ Task 2 (Queued)  ]  <-- Will finish
[ Task 3 (New)     ]  --X-- REJECTED
```

<!-- TOC --><a name="shutdownnow"></a>
### `shutdownNow()`
This is the "Emergency Exit." It tells the ES: "Stop everything immediately and go home!"

* **Behavior:** It tries to interrupt running tasks and returns a list of tasks that were waiting in the queue but never started.
* **Analogy:** A fire drill. Everyone drops what they are doing and runs out of the building immediately.

```text
[ Task 1 (Running) ]  <-- Interrupted!
[ Task 2 (Queued)  ]  <-- Returned to you in a List
```

<!-- TOC --><a name="awaittermination"></a>
###  `awaitTermination(...)`
Since `shutdown()` is asynchronous (it returns immediately while tasks finish in the background), you use `awaitTermination` to make your main thread wait.

* **Behavior:** It blocks your code for a specific amount of time (e.g., 5 seconds) or until all tasks are done.
* **Analogy:** Waiting outside the shop for your friend to finish their purchase after the "Open" sign is turned off.

<!-- TOC --><a name="isshutdown-vs-isterminated"></a>
### `isShutdown()` vs `isTerminated()`
These are status checks to see what phase the department is in.

* **`isShutdown()`**: Has the "Open" sign been turned off? (True if `shutdown()` was called).
* **`isTerminated()`**: Has every single person left the building and the lights are out? (True only when all tasks are finished).

```text
[ Running ] --shutdown()--> [ Shutting Down ] --(Tasks finish)--> [ Terminated ]
   ^ isShutdown=false           ^ isShutdown=true                 ^ isTerminated=true
```

| Method | New Tasks? | Existing Tasks? | Waiting? |
| :--- | :--- | :--- | :--- |
| **`submit()`** | Yes | Runs them | No (returns Future) |
| **`shutdown()`** | **No** | Finish them | No (returns immediately) |
| **`shutdownNow()`** | **No** | Kill them | No (returns List) |
| **`awaitTermination()`**| N/A | N/A | **Yes** (Blocks current thread) |

<!-- TOC --><a name="auto-closeable"></a>
## Auto-closeable

At its heart, **AutoCloseable** is a "contract" introduced in Java 7 to ensure that resources (like database connections, file streams, or network sockets) are closed automatically when they are no longer needed.

Before this, developers had to manually close resources in a `finally` block, which was famous for causing "Resource Leaks" if someone forgot to write the code.

<!-- TOC --><a name="the-try-with-resources-magic"></a>
### The "Try-With-Resources" Magic

To use **AutoCloseable**, you use a special syntax called **Try-With-Resources**. Any class that `implements AutoCloseable` can be put inside the parentheses of a `try` statement.

```text
1. [ Open Resource ]  (Inside try parentheses)
2. [ Run Code ]       (Inside try block)
3. [ Auto-Close ]     (JVM calls .close() automatically)
```

<!-- TOC --><a name="code-example-old-vs-new"></a>
### Code Example: Old vs. New

**The "Old" Way (Manual & Risky)**

You had to pray you didn't hit an exception before reaching the `close()` call.
```java
FileInputStream fis = null;
try {
    fis = new FileInputStream("file.txt");
    // Work with file
} catch (IOException e) {
    // Handle error
} finally {
    if (fis != null) fis.close(); // Very easy to forget!
}
```

**The "New" Way (AutoCloseable)**

The JVM guarantees `close()` is called, even if an exception happens.
```java
// Because FileInputStream implements AutoCloseable, we do this:
try (FileInputStream fis = new FileInputStream("file.txt")) {
    // Work with file
} // <--- fis.close() happens HERE automatically!
catch (IOException e) { ... }
```

**Creating Your Own AutoCloseable**

You can make your own classes "auto-closeable" by implementing the interface and providing a `close()` method.

```java
public class MyResource implements AutoCloseable {
    @Override
    public void close() {
        System.out.println("Cleaning up my custom resource!");
    }
}
```

**Why this matters for Virtual Threads**

This is a crucial link: **`ExecutorService` now implements `AutoCloseable` (since Java 19).**

This is why we can use the "New Pattern" for Virtual Threads:
```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(task);
} // <--- This calls executor.close(), which internally calls shutdown() and awaitTermination()
```

<!-- TOC --><a name="summary-comparison"></a>
### Summary Comparison

| Feature | Manual Closing | AutoCloseable (Try-with-resources) |
| :--- | :--- | :--- |
| **Safety** | Low (Leaks are common) | **High** (Guaranteed by JVM) |
| **Readability**| Messy (Finally blocks) | **Clean** (Parentheses syntax) |
| **Error Handling**| Complex | **Simple** (Suppressed exceptions handled) |

**Layman Definition:** It's like a "Self-Cleaning" oven. You put the food in, and when it's done, the oven cleans itself so you don't have to remember to do it later.

## Futures

A **Future** is a "placeholder" or a "receipt" for a result that hasn't happened yet. Imagine ordering a pizza: you pay at the counter, they give you a **receipt** (the Future), and you go sit down. You can't eat the receipt, but eventually, you exchange it for the actual pizza.

<!-- TOC --><a name="what-are-futures-simplest-example"></a>
### What are Futures? (Simplest Example)

In Java, a `Future<T>` represents the result of an asynchronous computation.

```java
// Calling a task that takes time
Future<String> receipt = executor.submit(() -> {
    Thread.sleep(2000); // Mimic work
    return "Fresh Pizza!"; 
});

// Do other things here...

// Now get the result (this blocks until it's ready)
String pizza = receipt.get(); 
```

<!-- TOC --><a name="futures-virtual-threads-the-perfect-match"></a>
### Futures + Virtual Threads: The "Perfect Match"

In the old days, calling `.get()` on a Future was dangerous because it **blocked** a precious Platform Thread. 

With **Virtual Threads**, blocking is "cheap." When a Virtual Thread calls `.get()` and the result isn't ready, it simply **unmounts** from the carrier thread. The CPU stays busy with other tasks while that specific VT waits in the heap.

```text
[ Main Thread ] ---submit task---> [ Virtual Thread (Worker) ]
      |                                  |
      | <--- returns Future (Receipt)    |-- Working... --|
      |                                  |
[ Main calls .get() ] <--- (Unmounts)    |-- Done!      --|
      |                                  |
[ Result Received ] <----- (Remounts) ---|
```

<!-- TOC --><a name="handling-microservice-responses"></a>
### Handling Microservice Responses

When calling an external API, the return type is usually a DTO (Data Transfer Object). We use `Callable` (which returns a value) instead of `Runnable`.

```java
public record UserProfile(String name, String email) {}

// Inside your service...
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    
    Future<UserProfile> futureProfile = executor.submit(() -> {
        // Simulating a REST call to a microservice
        return httpClient.get("http://auth-service/users/1", UserProfile.class);
    });

    // Handle the result
    try {
        // .get() with a timeout is a SENIOR/STAFF best practice!
        UserProfile profile = futureProfile.get(5, TimeUnit.SECONDS);
        System.out.println("User: " + profile.name());
    } catch (TimeoutException e) {
        System.err.println("Microservice took too long!");
    }
}
```

### Common Future Methods

| Method | Purpose |
| :--- | :--- |

| **`get(long, Unit)`** | Blocks, but throws `TimeoutException` if it takes too long. |
| **`isDone()`** | Returns `true` if the task finished (successfully or not). |

| :--- | :--- | :--- |

* In **Node.js**, `await` is mandatory because the single thread **must never block**. 
* In **Java (with VTs)**, blocking with `.get()` is totally fine because we have millions of threads. We don't need to worry as much about "Callback Hell."

<!-- TOC --><a name="using-semaphores-to-limit-concurrency-in-virtual-threads"></a>
## Using Semaphores to limit concurrency in Virtual Threads

When you use Virtual Threads (VTs), your application can easily spawn millions of concurrent tasks. However, the external world—like a downstream payment API or a legacy database—cannot handle millions of simultaneous connections. 

If a service limits you to, say, 5 concurrent requests, you might be tempted to use a `FixedThreadPool(5)`. But as we know, **Virtual Threads should never be pooled**. 

So, how do we decouple the creation of tasks from the limit of the external resource? **The answer is a `Semaphore`.**

<!-- TOC --><a name="what-is-a-semaphore-the-bouncer-analogy"></a>
### What is a Semaphore? (The "Bouncer" Analogy)

Think of a `Semaphore` as a bouncer at a very exclusive club (the downstream service). 
* The bouncer holds a specific number of **permits** (e.g., 5 VIP passes).
* When a Virtual Thread wants to enter, it must `acquire()` a permit.
* If all 5 permits are taken, the 6th Virtual Thread must wait in line.
* When a thread leaves the club, it must `release()` its permit back to the bouncer, allowing the next thread in line to enter.

```text
[ 10,000 Virtual Threads Spawned ]
        |
        v
  =========================================
  ||     SEMAPHORE (Max 5 Permits)       ||  <-- The Bouncer
  =========================================
        | (Only 5 VTs allowed past this point at a time)
        v
 [ VTs waiting politely in the Java Heap ]
        |
        v
[ DOWNSTREAM SERVICE (Rate Limited API) ]
```

<!-- TOC --><a name="why-is-this-safe-for-virtual-threads"></a>
### Why is this safe for Virtual Threads?

When a Virtual Thread calls `semaphore.acquire()` and no permits are available, **it does not block an OS Carrier Thread**. Instead, the VT elegantly "unmounts" and stores itself in the heap. 

This means you can have 10,000 Virtual Threads waiting at the Semaphore, and they will consume almost zero CPU and very little memory while they wait.

<!-- TOC --><a name="the-code-implementation"></a>
### The Code Implementation

Here is how you implement this pattern. We will spawn 15 Virtual Threads, but limit the downstream calls to exactly 3 at a time.

```java
import java.util.concurrent.Executors;
import java.util.concurrent.Semaphore;

public class VirtualThreadSemaphore {

    // 1. Create a Semaphore with exactly 3 permits
    private static final Semaphore bouncer = new Semaphore(3);

    public static void main(String[] args) {
        
        // 2. Use the standard Thread-Per-Task Executor (NO POOLING!)
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            
            // 3. Spawn 15 Virtual Threads almost instantly
            for (int i = 1; i <= 15; i++) {
                int taskId = i;
                executor.submit(() -> callDownstreamService(taskId));
            }
        } 
        // Executor automatically waits for all tasks to finish here
        System.out.println("All tasks completed.");
    }

    private static void callDownstreamService(int taskId) {
        try {
            // 4. Thread asks the bouncer for a permit. 
            // If none are available, the VT unmounts and waits here.
            bouncer.acquire(); 
            
            System.out.println("Task " + taskId + " ENTERED the service.");
            
            // Simulating network I/O to the limited service (takes 2 seconds)
            Thread.sleep(2000); 
            
            System.out.println("Task " + taskId + " FINISHED.");
            
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            // 5. CRITICAL: Always release the permit in a finally block!
            // This gives the ticket back so the next VT can enter.
            bouncer.release(); 
        }
    }
}
```

If you run the code above, you will see the Semaphore beautifully throttling your Virtual Threads. The output will happen in "batches" of 3, every 2 seconds.

**Output:**
```text
Task 1 ENTERED the service.
Task 2 ENTERED the service.
Task 3 ENTERED the service.
(.... pauses for 2 seconds ....)
Task 1 FINISHED.
Task 3 FINISHED.
Task 2 FINISHED.
Task 4 ENTERED the service.
Task 5 ENTERED the service.
Task 6 ENTERED the service.
(.... pauses for 2 seconds ....)
Task 5 FINISHED.
...and so on...
```

| Feature | Using a Fixed Thread Pool | Using VTs + Semaphore |
| :--- | :--- | :--- |
| **Code Structure** | Your application design is dictated by the slowest downstream API. | Your application is highly concurrent; only the specific API call is throttled. |
| **Queueing** | Tasks queue up outside the executor. | Tasks start instantly, but queue at the exact point of resource access. |
| **Granularity** | Hard to limit *just* the API call. | You wrap *only* the specific downstream call inside the `acquire/release` block. |

By using a `Semaphore`, you get to keep the "Thread-Per-Task" simplicity of Virtual Threads, while respecting the physical limits of the APIs and databases you depend on.

<!-- TOC --><a name="completable-futures"></a>
## Completable Futures

`CompletableFuture` is a **class** that implements `Future<T>`
- Provides many methods to chain, handle, error, combine results, etc
- We can also extend the `CompletableFuture` class!
- It by default uses the "Fork-join-pool"
- It accepts an `Executor`: We can use the "virtual-thread-per-task" executor

If a **Future** is a paper receipt, a **CompletableFuture (CF)** is a smart tracking app. It doesn't just wait for a result; it can trigger a chain of actions ("If the pizza is ready, put it in a box, then call the customer").

In the context of **Virtual Threads**, CF becomes an incredibly powerful orchestration tool for microservices.

<!-- TOC --><a name="practical-use-case-better-error-and-timeout-handling"></a>
### Practical use-case: Better error and timeout handling

`CompletableFuture` allows you to chain timeouts and error fallbacks as a clean, fluent pipeline instead of using messy `try-catch` blocks, while Virtual Threads ensure the underlying worker thread unmounts safely while waiting.

**The Code Comparison**

The Old Way (Future + Executor): *Clunky error handling and blocks the OS thread.*
```java
Future<String> future = executor.submit(() -> billingSvc.pay());

try {
    // Blocks the thread. Throws checked exceptions you MUST handle.
    String result = future.get(2, TimeUnit.SECONDS); 
} catch (TimeoutException | ExecutionException | InterruptedException e) {
    return "Payment Failed (Fallback)"; 
}
```

The New Way (CompletableFuture + Virtual Threads): *Clean pipeline, thread unmounts.*
```java
// supplyAsync + executor ensures Virtual Threads are used
String result = CompletableFuture.supplyAsync(() -> billingSvc.pay(), executor)
    .orTimeout(2, TimeUnit.SECONDS)               // Declarative timeout
    .exceptionally(ex -> "Payment Failed")        // Declarative fallback
    .join();                                      // Safely unmounts Virtual Thread
```

To put it in a broader system design perspective, ***this combination prevents cascading failures across your entire architecture***. 
- In the old model, if the downstream Billing Service suddenly became slow, all your heavy Platform Threads would get stuck waiting at that "wall," meaning your server would run out of resources and couldn't process any new requests—even for completely unrelated tasks like browsing the catalog
- By using `CompletableFuture` to enforce a strict timeout pipeline, combined with Virtual Threads that instantly free up the physical Carrier Thread while waiting, your application remains highly responsive under heavy load and simply returns a graceful fallback to the user instead of taking down the whole system.

```text
[ Old Way: The "Wall" ]
  Main Thread -> wait future.get(2s) --> [TIMEOUT WALL!] --> (Crash/Catch block)
                     (Pinned)

[ New Way: The "Pipeline" ]
  Virtual Thread -> wait API --> [2s Timeout] --> exceptionally() --> [Fallback Result]
                  (Unmounted)      (Routes error cleanly)              (Success)
```

<!-- TOC --><a name="starting-the-task-runasync-vs-supplyasync"></a>
### Starting the Task: `runAsync` vs `supplyAsync`

Both start a task in the background. The difference is whether you care about the return value.

**`runAsync()` (Fire and Forget)**

Used for tasks that don't return data, like logging or sending a metric.
* **JS Equivalent:** A promise that resolves to `undefined`.

```java
// Example: Logging a request to a telemetry service
CompletableFuture.runAsync(() -> {
    telemetrySvc.logRequest("Order Processed");
}, executor); // Always pass your Virtual Thread Executor!
```

**`supplyAsync()` (The Data Fetcher)**

Used when you need a result back (like a response from a Shipping service).
* **JS Equivalent:** `new Promise((resolve) => resolve(data))`.

```java
// Example: Fetching shipping status
CompletableFuture<String> statusFuture = CompletableFuture.supplyAsync(() -> {
    return shippingSvc.getStatus(orderId);
}, executor);
```

<!-- TOC --><a name="chaining-thencombine"></a>
### Chaining: `thenCombine()`

This is used when you need the results of **two independent** calls to perform a third action.

* **Microservice Example:** You need data from the **Price Service** AND the **Inventory Service** to show a product page.
* **JS Equivalent:** Using `Promise.all([p1, p2])` and then destructuring the results.

```java
CompletableFuture<Double> price = CompletableFuture.supplyAsync(() -> priceSvc.get(id), executor);
CompletableFuture<Integer> stock = CompletableFuture.supplyAsync(() -> invSvc.get(id), executor);

// Combine them when BOTH are ready
CompletableFuture<ProductDetail> detail = price.thenCombine(stock, (p, s) -> {
    return new ProductDetail(p, s);
});
```

<!-- TOC --><a name="advanced-orchestration-allof-and-anyof"></a>
### Advanced Orchestration: `allOf()` and `anyOf()`

**`allOf()` (The Barrier)**

Waits for **every** future in a list to finish. It returns `CompletableFuture<Void>`, so you usually follow it with a join.
* **JS Equivalent:** `Promise.all()`.

```java
// Calling 5 different microservices for a Dashboard
CompletableFuture<Void> allTasks = CompletableFuture.allOf(f1, f2, f3, f4, f5);

// Block until everyone is done
allTasks.join(); 
```

**`anyOf()` (The Race)**

Returns as soon as **any** one of the tasks finishes.
* **JS Equivalent:** `Promise.any()` or `Promise.race()`.
* **Microservice Example:** Calling three different "Mirror" databases; you just want the fastest response.

```java
CompletableFuture<Object> fastest = CompletableFuture.anyOf(db1Future, db2Future, db3Future);
Object result = fastest.join();
```

<!-- TOC --><a name="error-handling-exceptionally"></a>
### Error Handling: `exceptionally()`

In a microservice world, services fail constantly. CF lets you define a "Fallback" or "Default" if a call crashes.

* **Microservice Example:** If the **Recommendation Service** fails, show "Popular Items" instead of an error page.
* **JS Equivalent:** `.catch()`.

```java
CompletableFuture<List<String>> recs = CompletableFuture.supplyAsync(() -> recSvc.get(userId), executor)
    .exceptionally(ex -> {
        System.err.println("Rec Service Down: " + ex.getMessage());
        return List.of("Default Item 1", "Default Item 2"); // Fallback
    });
```

<!-- TOC --><a name="timeouts-ortimeout"></a>
### Timeouts: `orTimeout()`

Crucial for Senior/Staff engineers! Never wait forever for a microservice. Java 9+ added native timeout support to CF.

* **Behavior:** If the task isn't done in 2 seconds, throw a `TimeoutException`.
* **JS Equivalent:** Usually requires a `Promise.race` with a `setTimeout` timer.

```java
CompletableFuture<String> response = CompletableFuture.supplyAsync(() -> billingSvc.pay(), executor)
    .orTimeout(2, TimeUnit.SECONDS)
    .exceptionally(ex -> "Payment Timeout - Please retry");
```

<!-- TOC --><a name="the-extractor-join"></a>
### The Extractor: `join()`

Crucial for engineers to know the difference between `.get()` and `.join()`. While both do the same fundamental job (getting the final value), `join()` is the modern standard for extracting data from a CompletableFuture pipeline.

**Behavior**: It blocks the current thread until the result is ready. (Remember: if called on a Virtual Thread, it safely unmounts instead of wasting CPU!). Unlike `.get()`, which forces you to write messy try-catch blocks for checked exceptions, `.join()` throws an unchecked `CompletionException`, making your code much cleaner.

JS Equivalent: The `await` keyword.

<!-- TOC --><a name="summary-comparison-java-cf-vs-js-promises"></a>
### Summary Comparison: Java CF vs. JS Promises

| Java CompletableFuture | JavaScript Promise | Why it's different |
| :--- | :--- | :--- |
| `supplyAsync()` | `new Promise()` | Java explicitly separates "Run" from "Result". |
| `thenApply()` | `.then()` | Transforms the result for the next step. |
| `thenCombine()` | `Promise.all().then()` | Java makes combining two specific futures very readable. |
| `exceptionally()` | `.catch()` | Java provides a clean functional way to provide fallbacks. |
| `join()` | `await` | `join()` blocks the thread (Safe with Virtual Threads!). |

<!-- TOC --><a name="why-use-cf-with-virtual-threads"></a>
### Why use CF with Virtual Threads?
While Virtual Threads make standard blocking code (`future.get()`) safe, **CompletableFuture** is still better for **Parallelism**. 

If you call 3 services sequentially with `get()`, it takes 3x the time. If you use `allOf()` or `thenCombine()` with a Virtual Thread executor, they run **at the same time**, and the Virtual Threads will unmount while waiting for the network, giving you the best of both worlds: **High Concurrency and Low Latency.**

<!-- TOC --><a name="competablefuture-and-virtual-threads"></a>
## CompetableFuture and Virtual Threads

The combination of **CompletableFuture (CF)** and **Virtual Threads (VT)** is the "Gold Standard" for modern Java backend architecture. 

In the old days (Platform Threads), using CF was often a nightmare because you had to manage complex thread pools (`ForkJoinPool.commonPool()`) and worry about blocking those precious threads. With Virtual Threads, that "fear of blocking" disappears.

<!-- TOC --><a name="the-safety-factor-cheap-blocking"></a>
### The "Safety" Factor: Cheap Blocking

The biggest reason CF is safe with VTs is that **`.join()`** or **`.get()`** are no longer "scary" operations.

* **In Platform Threads:** If you call `.join()`, a 1MB OS thread sits idle. If 1,000 users do this, your server runs out of memory.
* **In Virtual Threads:** If you call `.join()`, the Virtual Thread **unmounts**. The physical Carrier Thread is freed up to do other work.

```text
[ PLATFORM THREAD ]                [ VIRTUAL THREAD ]
|  Running Task   |                |  Running Task  |
|  Calls .join()  |                |  Calls .join() |
|  [ BLOCKED ]    | <--- Waste     |  -- UNMOUNT -- | <--- Efficiency
|  (Still 1MB)    |                |  (Moved to Heap)
```

<!-- TOC --><a name="example-parallel-microservice-orchestration"></a>
### Example: Parallel Microservice Orchestration

Imagine a **Travel Booking Service** that needs to fetch Flight, Hotel, and Car Rental details. You want to do these in parallel to save time.

```java
public BookingDetails getBooking(String userId) {
    // 1. Define the Executor (The Virtual Thread Factory)
    try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {

        // 2. Start tasks in parallel using supplyAsync
        CompletableFuture<Flight> flightReq = CompletableFuture.supplyAsync(() -> 
            flightSvc.find(userId), executor);
            
        CompletableFuture<Hotel> hotelReq = CompletableFuture.supplyAsync(() -> 
            hotelSvc.find(userId), executor);

        // 3. Orchestrate with allOf (Safety: this doesn't hog OS threads)
        CompletableFuture.allOf(flightReq, hotelReq).join(); 

        // 4. Return combined result
        return new BookingDetails(flightReq.join(), hotelReq.join());
    }
}
```

**Why `supplyAsync` needs the `executor` argument**

By default, `CompletableFuture` uses the **Common ForkJoinPool**, which uses a small number of Platform Threads (usually equal to your CPU cores). 
- **CRITICAL RULE:** To make CF "Safe" and scalable with Virtual Threads, you **must** pass your Virtual Thread executor as the second argument. 
  * **Bad:** `CompletableFuture.supplyAsync(() -> task())` // Uses limited Platform Threads.
  * **Good:** `CompletableFuture.supplyAsync(() -> task(), executor)` // Uses unlimited Virtual Threads.


<!-- TOC --><a name="5-comparison-java-cfvt-vs-nodejs"></a>
### 5. Comparison: Java CF/VT vs. Node.js
If you are used to Node.js `Promise.all()`, notice the difference in "Safety":

| Feature | Node.js (Async/Await) | Java (CF + VT) |
| :--- | :--- | :--- |
| **Concurrency** | Cooperative (Event Loop). | Cooperative (JVM Scheduler). |
| **Safety** | One long task blocks everyone. | One long task only blocks 1 VT. |
| **Syntax** | `await Promise.all(...)` | `CompletableFuture.allOf(...).join()` |
| **Scaling** | Limited by single-thread CPU. | Scales to millions of cores/threads. |

<!-- TOC --><a name="drawbacks-of-completablefuture"></a>
### Drawbacks of CompletableFuture

Calling `.cancel()` on a CompletableFuture only cancels the receipt (the future itself) but ***explicitly does not interrupt the underlying thread***, meaning the background task keeps running and wasting resources.

```java
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    try {
        Thread.sleep(5000); // Simulating a long API call
        // THIS WILL STILL PRINT, even after cancel() is called!
        System.out.println("Task finished, but nobody is listening..."); 
    } catch (InterruptedException e) {
        System.out.println("Interrupted!"); // CF never triggers this.
    }
    return "Data";
}, executor);

// The 'true' argument is explicitly IGNORED by CompletableFuture.
// The Future throws a CancellationException if you call .join(), 
// but the Virtual Thread above keeps running in the background.
future.cancel(true);
```

```
[ Main Thread ] --cancel()--> [ Future Receipt: CANCELLED ]
                                      | (Link severed)
[ Virtual Thread ] -------------------+-----> [ Keeps running... Wastes CPU/IO ]
```

<!-- TOC --><a name="threadlocal-scoped-variables"></a>
## ThreadLocal & Scoped variables

Imagine you walk into a massive gym. If everyone had to share the exact same gym bag (a global variable), it would be chaos—people would constantly overwrite or steal each other's gear. 

**`ThreadLocal`** is like giving every single person who walks into the gym their own private, invisible locker. Everyone accesses their locker using the exact same key (the variable name), but the gym magically ensures they only see *their own* stuff.

Here is the deep dive into `ThreadLocal`—a concept that is absolutely critical for understanding how modern backend frameworks (like Spring Boot) manage concurrency and context.

<!-- TOC --><a name="what-is-threadlocal"></a>
### What is `ThreadLocal`?
Introduced way back in **Java 1.2**, `ThreadLocal` is a special class that allows you to create variables that can only be read and written by the **same thread** that created them. 

Even if two threads are executing the exact same line of code, and looking at the exact same `ThreadLocal` instance, they will see completely different values.

<!-- TOC --><a name="ascii-diagram-the-mental-model"></a>
#### ASCII Diagram: The Mental Model

```text
[ Global Variable ]                  [ ThreadLocal Variable ]
  "Shared state"                       "Private state"

    [ Thread 1 ] \                        [ Thread 1 ] -> Value A
                  \                      /
    [ Thread 2 ] --- [ Value ]  <--->  [ ThreadLocal ] -> Value B
                  /                      \
    [ Thread 3 ] /                        [ Thread 3 ] -> Value C
```

<!-- TOC --><a name="why-do-we-need-it-the-problem"></a>
### Why do we need it? (The Problem)

Imagine a standard web application. A user sends an HTTP request. The server assigns a Thread from its Thread Pool to handle it. 

During this request, you generate a unique `Transaction ID` so you can track it through your logs. Your code path looks like this:
`Controller -> AuthenticationService -> BusinessLogic -> DatabaseRepository -> NotificationService`

**The Bad Way (Parameter Pollution):**
You have to pass `transactionId` into every single method signature, even if `BusinessLogic` doesn't care about it.
```java
// Ugly and hard to maintain
public void doWork(User user, String txId) {
    db.save(user, txId);
    notifier.send(user, txId); 
}
```

**The Dangerous Way (Global Variable):**
You use a `static String transactionId`. But if 5,000 users hit your server at once, Thread A overwrites Thread B's ID. Race conditions explode.

**The `ThreadLocal` Way:**
You store the ID in a `ThreadLocal`. Now, any method, anywhere in the call stack, can ask the current thread: *"Hey, what is MY transaction ID?"* without needing it passed as a parameter.

<!-- TOC --><a name="how-to-use-it-the-examples"></a>
### How to use it? (The Examples)

A `ThreadLocal` is usually declared as `private static final`. Here is how you build a "Context Holder" for user requests.

<!-- TOC --><a name="example-1-the-basic-api"></a>
#### Example 1: The Basic API
```java
public class UserContext {
    // 1. Create the ThreadLocal
    private static final ThreadLocal<String> currentUser = new ThreadLocal<>();

    // 2. Set the value (usually done in an Interceptor or Filter at the start of the request)
    public static void setUser(String username) {
        currentUser.set(username);
    }

    // 3. Get the value (can be called deep inside a repository layer)
    public static String getUser() {
        return currentUser.get();
    }

    // 4. Remove the value (CRITICAL!)
    public static void clear() {
        currentUser.remove();
    }
}
```

<!-- TOC --><a name="example-2-in-action"></a>
#### Example 2: In Action
```java
public class Worker implements Runnable {
    private String name;
    public Worker(String name) { this.name = name; }

    @Override
    public void run() {
        UserContext.setUser(name);
        System.out.println(Thread.currentThread().getName() + " set user to: " + name);
        
        CommonUtils.sleep(100); // Simulate some work
        
        // Notice we didn't pass 'name' to this method! It magically knows.
        printContext(); 
    }

    private void printContext() {
        System.out.println(Thread.currentThread().getName() + " sees user: " + UserContext.getUser());
    }
}

// Result:
// Thread-1 set user to: Alice
// Thread-2 set user to: Bob
// Thread-1 sees user: Alice
// Thread-2 sees user: Bob
```

<!-- TOC --><a name="benefits"></a>
### Benefits
* **Thread Safety without Synchronization:** Because threads never share the data, you completely avoid using `synchronized` blocks or locks, meaning zero performance bottlenecks.
* **Cleaner Code:** It prevents "Parameter Drilling" (passing arguments down 10 layers of methods).
* **Framework Power:** This is exactly how Spring Security stores the logged-in user (`SecurityContextHolder`), how SLF4J/Logback stores log tracing IDs (`MDC`), and how databases manage active Transactions.

<!-- TOC --><a name="the-golden-rule-best-practices"></a>
### The Golden Rule & Best Practices

**THE GOLDEN RULE: You MUST call `.remove()` in a `finally` block.**

Why? Because modern servers (like Tomcat) use **Thread Pools**. Threads are not destroyed when a request finishes; they are returned to the pool for the next user. 

If you forget to clear the `ThreadLocal`, the next user who borrows that thread will see the previous user's data. This causes **massive security breaches** and **Memory Leaks** (because the garbage collector cannot clean up the object while the thread is still alive and holding a reference to it).

```java
// Best Practice Implementation
public void handleRequest(Request req) {
    try {
        UserContext.setUser(req.getUserId());
        // ... execute 100 layers of business logic ...
    } finally {
        UserContext.clear(); // NEVER FORGET THIS
    }
}
```

<!-- TOC --><a name="ascii-diagram-the-thread-pool-leak"></a>
#### ASCII Diagram: The Thread Pool Leak
```text
[ Request 1 (Alice) ] -> Borrows Thread 1 -> Sets ThreadLocal="Alice"
[ Request 1 Ends ]    -> Thread 1 returns to pool (Still holding "Alice"!)

[ Request 2 (Bob) ]   -> Borrows Thread 1 -> Asks for User... 
[ DANGER! ]           -> Returns "Alice" instead of Bob!
```

<!-- TOC --><a name="when-not-to-use-it"></a>
### When NOT to use it

1. **Passing Business Logic:** Do not use `ThreadLocal` to pass standard business data (like a `CartItem` or an `Order`). It makes the code incredibly hard to read and test because the data flow is invisible. It should only be used for "Cross-Cutting Concerns" (Security, Logging, Transactions).
2. **Asynchronous Code / CompletableFutures:** If your main thread spawns a background thread (e.g., using `CompletableFuture.supplyAsync()`), **the background thread does NOT inherit the `ThreadLocal` data.** The context is lost. (You have to use specialized classes like `InheritableThreadLocal` or manually copy the context over).

<!-- TOC --><a name="the-modern-shift-java-21-and-virtual-threads"></a>
### The Modern Shift: Java 21+ and Virtual Threads

If you are designing a modern, highly scalable backend, you need to know how `ThreadLocal` interacts with **Virtual Threads**.

Because Virtual Threads are so cheap, a server might have 1,000,000 of them running at once. If your application relies heavily on `ThreadLocal` (storing big objects), and you spawn a million virtual threads, you will instantly create 1,000,000 copies of that data in memory, causing a massive heap bloat.

**The Future: Scoped Values**
In modern Java (previewed in 21, maturing in 24), a new feature called **Scoped Values** (`ScopedValue<T>`) is being introduced to replace `ThreadLocal`. 
* They are **immutable** (you can't change them halfway through a request).
* They automatically clean themselves up (no memory leaks, no `.remove()` required).
* They share data efficiently across thousands of Virtual Threads without duplicating memory. 

Until Scoped Values become the absolute industry standard, `ThreadLocal` remains the undisputed king of context management in Java backends—provided you always remember to clean out your locker before leaving the gym.

<!-- TOC --><a name="scoped-values"></a>
## Scoped Values

If `ThreadLocal` is like assigning every person their own permanent locker at the gym, a **`Scoped Value`** is like handing a person a VIP wristband that automatically dissolves the moment they walk out the door.

For Senior and Staff-level architecture, especially when designing for millions of concurrent Virtual Threads, `ThreadLocal` is becoming a massive liability. **Scoped Values** are Java’s modern answer to this problem.

Here is the deep dive into what they are, how they work, and why you (as a Senior Engineer) should absolutely know them, but explicitly **not** put them in production just yet.

<!-- TOC --><a name="why-do-we-need-them-the-threadlocal-disaster"></a>
### Why do we need them? (The `ThreadLocal` Disaster)

With the release of Java 21, **Virtual Threads** allow us to run millions of concurrent tasks. However, Virtual Threads and `ThreadLocal` are a dangerous combination.

* **The Memory Bloat:** If a `ThreadLocal` stores a 1MB User Context object, and you spawn 1,000,000 Virtual Threads, you just consumed 1 Terabyte of RAM.
* **The Mutability Problem:** Any piece of code can call `ThreadLocal.set()`. Deep inside a 50-layer call stack, a rogue method could overwrite the User ID, causing impossible-to-find bugs.
* **The Leak Problem:** As discussed previously, if you forget `finally { remove() }`, the data leaks.

<!-- TOC --><a name="what-are-scoped-values"></a>
### What are Scoped Values?

A **Scoped Value** is a way to share data safely and efficiently across a specific block of code (a "scope") without passing it as a method parameter. 

They have three golden properties:
1. **Immutable:** Once set for a block of code, the value cannot be changed.
2. **Bounded Lifetime:** The value only exists for the duration of the `run()` block. When the block ends, the value is destroyed instantly.
3. **Incredibly Cheap:** They are highly optimized for Virtual Threads. They do not duplicate memory heavily across child threads.

```text
[ ThreadLocal ] - The Infinite Line
  Start -> set("Alice") ------------------> get() --------------------> (Leaked forever if no remove())
                                                                        |
[ ScopedValue ] - The Bounded Box                                       v
  Start -> [ where(USER, "Alice").run(...) ]                      (Disaster)
                 |
                 +-> Method A calls get() -> "Alice"
                 +-> Method B calls get() -> "Alice"
                 |
           [ Block Ends -> Value Auto-Destroys! ] --> Safe!
```

<!-- TOC --><a name="how-to-use-scoped-values-code-examples"></a>
### How to Use Scoped Values (Code Examples)

You define a Scoped Value similarly to a `ThreadLocal`, usually as a `public static final` field.

<!-- TOC --><a name="step-1-declaration"></a>
#### Step 1: Declaration
```java
public class ContextHolder {
    // Defines the key, but holds no value yet.
    public static final ScopedValue<String> LOGGED_IN_USER = ScopedValue.newInstance();
}
```

<!-- TOC --><a name="step-2-execution-the-where-run-pattern"></a>
#### Step 2: Execution (The "Where / Run" Pattern)
Instead of setting a value and hoping it gets cleaned up, you explicitly define the boundaries of where the value is valid.

```java
public void handleRequest(String username) {
    
    // "Where LOGGED_IN_USER is 'username', run this block of code"
    ScopedValue.where(ContextHolder.LOGGED_IN_USER, username)
               .run(() -> {
                   // Inside this block, the value is active.
                   processPayment();
                   sendEmail();
               }); 
    
    // The millisecond the block finishes, the Scoped Value is gone. No finally block needed!
}
```

<!-- TOC --><a name="step-3-reading-the-value"></a>
#### Step 3: Reading the Value
Deep inside your microservice layers (e.g., inside `sendEmail()`), you can read the value:

```java
public void sendEmail() {
    // Check if it exists, then get it
    if (ContextHolder.LOGGED_IN_USER.isBound()) {
        String user = ContextHolder.LOGGED_IN_USER.get();
        System.out.println("Sending email to: " + user);
    }
}
```

<!-- TOC --><a name="best-practices-golden-rules"></a>
### Best Practices & Golden Rules

1. **Use for Cross-Cutting Concerns Only:** Just like `ThreadLocal`, use them for Security Contexts, Distributed Tracing IDs, or Tenant IDs in multi-tenant architectures. Do not use them to pass standard business objects.
2. **Embrace Immutability:** If you need to "update" a Scoped Value, you can't. You must nest a new scope inside the old one. 
    ```java
    ScopedValue.where(ROLE, "USER").run(() -> {
        // ... code ...
        // Re-bind temporarily for a specific admin task
        ScopedValue.where(ROLE, "ADMIN").run(() -> {
            // I am ADMIN here
        });
        // I am back to USER here
    });
    ```
3. **Pair with Structured Concurrency:** Scoped Values magically inherit down to child threads *if* you use the new `StructuredTaskScope` API. This solves the nightmare of passing `ThreadLocal` context into asynchronous background tasks.

<!-- TOC --><a name="why-they-are-too-early-for-prod-the-reality-check"></a>
### Why they are "Too Early for Prod" (The Reality Check)

If you are interviewing for a Staff Engineering role, you must know when **not** to adopt new shiny tech. As of Java 21, 22, and 23, Scoped Values are still a **Preview Feature**. 

Here is why you cannot use them in a production system like Google or a Tier 1 bank yet:

* **The `--enable-preview` Flag:** To run code with Scoped Values, you must start the JVM with the `--enable-preview` flag. In enterprise production environments, this flag is strictly forbidden. It means the feature is subject to change, and Oracle provides absolutely zero guarantees of backward compatibility in the next patch.
* **Framework Support is Zero:** You don't usually write raw `ThreadLocal` code; you rely on Spring Boot, Hibernate, Logback, and Tomcat to manage contexts. None of these major frameworks have migrated to Scoped Values yet because the API is not finalized. If you use a Scoped Value, your logging framework won't be able to see it for tracing.
* **API Instability:** The exact syntax (`where()`, `run()`, `call()`) has already shifted slightly during the preview phases. If you write production code on it today, a minor Java update could literally break your compilation.

<!-- TOC --><a name="the-summary-comparison"></a>
### The Summary Comparison

| Feature | `ThreadLocal` | `ScopedValue` |
| :--- | :--- | :--- |
| **Mutability** | Mutable (Anyone can change it) | **Immutable** (Read-only for the block) |
| **Lifecycle** | Unbounded (Must manually `remove()`) | **Bounded** (Auto-cleans at end of block) |
| **Memory Footprint** | Heavy (Copies per thread) | **Ultra-Light** (Optimized for Virtual Threads) |
| **Child Threads** | Heavy/Complex to inherit | **Native/Free** via Structured Concurrency |
| **Production Ready?**| Yes (Since 1998) | **NO** (Preview Feature as of Java 21/23) |

**The Final Verdict:** Learn them now so you can design architectures that will be ready for them in Java 25+. But until the `java.lang.ScopedValue` drops its "Preview" status and Spring Boot adopts it, keep using `ThreadLocal` with strict `finally { remove() }` blocks.
